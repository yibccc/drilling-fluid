# 冷热分离架构 - 面试话术

## 一句话总结

> "热数据放 Redis 保证实时性能，冷数据放 PostgreSQL 做持久化和查询，通过 Redis Stream 异步同步，实现读写分离和存储解耦。"

---

## 为什么需要冷热分离？

**业务场景特点：**
- 对话过程需要**低延迟**（流式响应，毫秒级）
- 历史查询需要**持久化**（用户随时查看过去记录）
- 数据访问有明显的**冷热特性**（活跃对话 vs 历史归档）

**如果只用一个存储的问题：**
- 只用 PostgreSQL：每次对话都写库，高并发下性能瓶颈
- 只用 Redis：数据易丢失，无法做复杂查询，内存成本高

---

## 为什么选择 Redis？

**选择理由：**

1. **性能匹配** - 对话场景需要频繁读写状态，Redis 内存操作毫秒级响应
2. **官方支持** - LangGraph 官方提供 `AsyncRedisSaver`，开箱即用 checkpoint 管理
3. **状态恢复** - 天然支持状态快照和恢复，适合对话中断后继续
4. **数据结构** - Hash/String 结构适合存储结构化状态数据

**AsyncRedisSaver 核心概念：**

在 LangGraph 中，内存分为两类：

| 类型 | 作用 | 存储位置 | 官方实现 |
|------|------|----------|----------|
| **短期记忆** | 跟踪多轮对话状态 | Checkpointer | AsyncRedisSaver |
| **长期记忆** | 跨会话存储用户数据 | Store | AsyncPostgresStore |

我们的项目中：
- **Redis (AsyncRedisSaver)** 存短期记忆 — 对话过程中的状态快照
- **PostgreSQL** 存长期记忆 — 历史消息和会话归档

**AsyncRedisSaver 工作原理：**

```
用户请求 (带 thread_id)
    ↓
graph.invoke({"messages": [...]}, {"configurable": {"thread_id": "session_123"}})
    ↓
AsyncRedisSaver 自动保存状态到 Redis
    ↓
下次请求使用相同 thread_id，自动恢复上下文
```

**为什么不选其他？**
- Memcached：无持久化，数据结构单一
- MongoDB：延迟高于 Redis，对 checkpoint 场景无优势

---

## 为什么选择 PostgreSQL？

**选择理由：**

1. **关系查询** - 需要按用户、时间等多维度查询会话列表
2. **数据安全** - 磁盘持久化，支持事务和备份
3. **JSONB 支持** - tool_calls 等复杂结构用 JSONB 存储，灵活且高性能
4. **成熟生态** - 团队熟悉，运维成本低

**为什么不选其他？**
- MySQL：JSON 支持不如 PostgreSQL 的 JSONB
- MongoDB：缺少事务保证，数据一致性要求高时更放心关系型数据库

---

## 为什么用 Redis Stream 同步？

**选择理由：**

1. **架构简单** - 不需要引入 Kafka/RabbitMQ 等中间件
2. **异步解耦** - 同步操作不阻塞用户对话，提升用户体验
3. **消息可靠** - Consumer Group + ACK 机制保证消息不丢失
4. **运维友好** - 项目本身就用 Redis，不增加额外组件

**同步时机：** 对话完成后触发，不需要实时同步（最终一致性）

---

## 怎么实现的？

### 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                      用户请求                               │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    ChatService                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Redis (热数据)                                      │   │
│  │  · 对话状态 - LangGraph checkpoint                   │   │
│  │  · 消息历史 - 实时读写，毫秒级响应                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                          │                                  │
│                          │ 对话完成后                        │
│                          ▼                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Redis Stream (同步触发)                              │   │
│  │  发送 session_id + message_count                      │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                    SyncService (后台)                        │
│  1. 消费 Redis Stream 消息                                   │
│  2. 从 Redis checkpoint 读取完整状态                         │
│  3. 写入 PostgreSQL (会话 + 消息)                            │
└─────────────────────────────────────────────────────────────┘
```

### 数据分层

| 数据类型 | 存储位置 | 访问频率 | 保留策略 |
|---------|---------|---------|---------|
| 活跃对话状态 | Redis | 极高 | 活跃期保留 |
| 历史消息 | PostgreSQL | 低 | 永久 |
| Checkpoint 备份 | PostgreSQL | 按需 | 永久 |

### Checkpoint 工作原理（LangGraph 核心）

**什么是 Checkpoint？**

Checkpoint 是 LangGraph 的**状态快照机制**，每次对话执行后自动保存当前状态到 Redis：

```
┌─────────────────────────────────────────────────────────────┐
│                   LangGraph 执行流程                       │
├─────────────────────────────────────────────────────────────┤
│  用户消息 → 处理节点 → 保存 Checkpoint → 返回响应       │
│                              ↓                        │
│                        Redis (AsyncRedisSaver)         │
│                        key: checkpoint_id              │
│                        val: {state, messages, metadata} │
└─────────────────────────────────────────────────────────────┘
```

**核心 API：**

| API | 作用 | 使用场景 |
|-----|------|---------|
| `graph.get_state(config)` | 获取最新状态 | 查看当前对话历史 |
| `graph.get_state_history(config)` | 获取历史快照 | 回溯对话状态 |
| `graph.invoke(..., config)` | 指定 thread_id 恢复对话 | 断点续传 |

**thread_id 的作用：**

```python
# 每个会话有独立的 thread_id，彼此隔离
config = {"configurable": {"thread_id": "session_123"}}

# 相同 thread_id 的请求会自动恢复历史上下文
graph.invoke({"messages": "我之前说啥了？"}, config)
# → 自动从 Redis 加载之前的对话历史
```

### 容错设计

**Redis 不可用时的降级：**
- 自动切换到 `MemorySaver`（内存存储）
- 服务继续可用，但重启后状态丢失
- 日志记录告警，人工介入

**同步失败重试：**
- Consumer Group 未 ACK 的消息会被重新消费
- 使用 `XCLAIM` 处理超时未完成的任务

## 面试追问应对

### Q: 数据不一致怎么办？
**A:** 采用最终一致性模型，对话场景允许秒级延迟。同步失败会重试，Consumer Group 保证消息至少处理一次。

### Q: Redis 数据全部丢失怎么恢复？
**A:** PostgreSQL 有 `langchain_checkpoints` 备份表，可以从这里恢复到 Redis。同时 Redis 本身配置了 RDB + AOF 持久化，全丢概率极低。

### Q: 为什么不直接写入 PostgreSQL？
**A:**
1. 高并发场景下数据库写入压力太大
2. 流式对话需要频繁更新状态，每次都写库响应慢
3. Redis 读写性能是数据库的 10-100 倍

### Q: 历史查询怎么优化？
**A:** 对 `chat_messages.session_id` 建索引，分页查询。如果需要全文检索，可以考虑接入 Elasticsearch。

---

## 关键亮点总结

1. **架构清晰** - 冷热分离职责明确
2. **技术选型合理** - 每个组件都有明确的业务价值
3. **异步解耦** - 同步操作不影响用户体验
4. **容错完善** - 降级 + 重试保证高可用
